# learning-test — память проекта

Набор обучающих тестов (GitHub Pages). Прогресс сохраняется в localStorage и
синхронизируется с Google Drive. Сайт **публикуется из ветки `main`** (GitHub
Pages) — чтобы изменения появились на сайте, они должны попасть в `main`.

## Архитектура

- `scripts/content/<slug>.json` — исходные данные теста: массив из **50** вопросов
  `{q, options[4], correct, explain, usage}`. `correct` — индекс правильного
  варианта (0..3). `usage` = `{code, without, with, benefit}` (пример «как без
  этого / как с этим»).
- `scripts/template.html` — шаблон страницы теста.
- `scripts/build.py` — вшивает `content/*.json` в шаблон → `tests/<slug>.html`.
  **После любой правки контента запускать `python3 scripts/build.py`.**
- `index.html` — список тестов; `sims/` — интерактивные лаборатории; `sync.js` —
  общая синхронизация всех тестов с Google Drive; `config.js` — GOOGLE_CLIENT_ID.
- Формат JSON контента — **компактный**: массив, по одному вопросу на строку,
  разделители без пробелов. Скрипты правок это сохраняют.

## Правила создания тестов (ОБЯЗАТЕЛЬНО)

Цель — чтобы правильный ответ нельзя было угадать по **форме**, а только по
знанию. Все правила проверяются `python3 scripts/check_quality.py` (код возврата
≠ 0 при нарушении). Запускать этот чек перед коммитом любого теста.

1. **Ровно 50 вопросов** в тесте, у каждого **ровно 4 варианта** — непустых и
   различных; `correct` в диапазоне 0..3.

2. **Позиция правильного ответа — случайная.** Правильный не должен
   систематически стоять на одной позиции (частая ошибка — почти всегда индекс 0
   или 1). Распределение по 0..3 должно быть ~равномерным (каждая позиция ≥ ~5
   раз из 50). Инструмент: `python3 scripts/shuffle_options.py` перемешивает
   варианты и пересчитывает `correct`.

3. **Правильный ответ НЕ должен быть самым длинным.** Нельзя, чтобы правильный
   опознавался по объёму текста. Требования: длины всех 4 вариантов сопоставимы;
   доля «правильный строго самый длинный» ≤ ~35% (в идеале ~25%); средняя длина
   правильного ≈ средней длине неправильных. Как достигать: правильный подрезать,
   где многословно; неправильные дополнять **правдоподобными, но фактически
   неверными** деталями до похожей длины (не превращать дистрактор в правду и не
   делать его перефразом правильного).

4. **В правильном ответе не должно быть выдающих круглых скобок `( )`.** Скобки
   часто ставят только в правильном (пояснения, формулы, аббревиатуры) — и это
   выдаёт ответ. Правило: **не должно быть вопросов, где скобка есть только у
   правильного варианта.** Способы: убрать скобку из правильного (перефразировать)
   ИЛИ добавить естественную скобку в один из **неправильных** вариантов (оставив
   его однозначно неверным). Правильный вариант при этом по смыслу не менять.

5. **Содержание.** Правильный остаётся фактически верным; неправильные —
   однозначно неверными и не перефразом правильного. `usage`, `q`, `explain` при
   косметических правках вариантов не трогать.

## Рабочий процесс правок контента

1. Правки в `scripts/content/*.json` (или через вспомогательные скрипты —
   `shuffle_options.py`, `apply_balanced.py`, `apply_parenfix.py`, `merge_usage.py`;
   они пишут компактный формат и содержат ассерты целостности).
2. `python3 scripts/check_quality.py` — все тесты должны пройти.
3. `python3 scripts/build.py` — пересобрать HTML.
4. Проверить в браузере (Playwright есть в окружении: модуль в
   `/opt/node22/lib/node_modules/playwright`, Chromium предустановлен).
5. Коммит в рабочую ветку; чтобы попало на сайт — слить в `main` (Pages
   пересоберётся автоматически).

## Полезное

- Разработка идёт в отдельной ветке; `main` — то, что задеплоено. Разрыв «на
  сайте старая версия» обычно значит, что последние коммиты не влиты в `main`.
- Проверка деплоя: workflow `pages build and deployment` (все запуски — с `main`).
- Модель раздачи GitHub Pages кэширует HTML; для проверки свежей версии —
  жёсткое обновление (Ctrl/Cmd+Shift+R) или приватное окно.
